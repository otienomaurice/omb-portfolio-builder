1. First setup: identity, branding, and contact details

Start here when the builder is blank or when you are setting up a new machine. These fields establish who owns the portfolio before projects are added.

  1. Open Profile and contact.
  2. Add display name, portfolio label, contact intro, email, phone, GitHub, LinkedIn, and website links.
  3. Add the profile image, brand/trademark image, resume PDF, and main page background.
  4. Check that email opens a compose window and phone uses a call link on supported devices.
  5. Click Save draft after the profile looks correct.

Good profile text is short and recruiter-facing. Put personality in Fun facts and personal sections, not in the contact labels.

2. Front page text and Fun facts

Front page text controls the first message recruiters see. Fun facts sit near the top of the site and make the portfolio feel human without interrupting the engineering content.

  1. Use the eyebrow as a short label, such as Hardware Engineering Portfolio.
  2. Use the main title for the strongest one-line identity statement.
  3. Use the supporting copy to explain what the portfolio contains.
  4. Keep Fun facts to one to three short lines.
  5. Save draft, then open Portfolio preview to verify the top of the page.

Example Fun facts: I built a builder to build this portfolio. I enjoy a good argument about soccer tactics, culture, and refereeing.

3. Project categories and project creation

All project creation starts from the main Add project control. Do not add projects from inside a category row.

  1. Click Add project.
  2. Choose the category or create a new category if the project does not fit the existing ones.
  3. Select appearance. Appearance controls visual style only; it must not insert fake project text.
  4. Choose placement: top or bottom of the selected category.
  5. Type a realistic project title, then create the project.

Only categories that contain projects show project rows. Empty categories stay as clean category headers until a project is added.

4. Project structure and parser rules

Projects are built by the parser from the sections you save. The website should show only valid, populated project content.

  1. New projects include Overview and Design by default.
  2. Additional sections such as Tests, Tools, PCB Builds, Documentation, Simulation, References, or Results must be added manually.
  3. Design and Simulation are separate major sections.
  4. An overview appears automatically when it exists; it is not shown as a separate clickable Overview icon.
  5. Empty sections, empty subsections, and empty files do not render on the website.

Use Save to parse the project and update its project preview. Use Save and close when you are done editing that project window.

5. Rich text editing and right-click menu

Rich text fields behave like document editors. Click where you want to type. Highlight exact text before applying formatting.

  1. Right-click highlighted text to copy, cut, paste, select all, change font, change size, change color, and apply bold, italic, or underline.
  2. The default typing font is Arial.
  3. If selected text has mixed font, size, or color, the inspector fields are blank so you know the selection is mixed.
  4. Use Enter for a new paragraph. Pasted line breaks remain line breaks.
  5. Typing and wrapping should not create fake new paragraphs; only hard enters and pasted line breaks do.

If the selection menu disappears, click back inside the rich editor and highlight the text again.

6. Links, formulas, and pasted content

The builder distinguishes normal links from mathematical formulas.

  1. URLs such as linkedin.com, github.com, www links, and https links are treated as links.
  2. Use Add mathematical formula only when you intentionally want a formula block.
  3. Links in rich text open as websites on the portfolio unless you attach them as downloadable files.
  4. PDFs, spreadsheets, code files, archives, and documents are saved as downloadable resources.
  5. Image files can be visible on the website or hidden as downloadable evidence.

When pasting from ChatGPT, review formulas before saving. If the content is a link, it should render as a link, not as a formula.

7. Images and file evidence

Add evidence where it belongs so the final portfolio reads like a technical record, not a file dump.

  1. Paste an image into a rich editor to place it below the cursor line.
  2. Use Add image when you need source selection, rename, caption, alignment, crop, or visible/download-only control.
  3. Drag an inserted image to move the same image; the editor should not duplicate it.
  4. Add PDFs under documents, lab reports, design notes, or references.
  5. Add simulation files under Simulation files and screenshots/plots under Results.
  6. Add source repositories as links, not downloadable text files.

Use titles that explain the evidence: Oscilloscope capture, KiCad PCB layout, LTspice transient result, VCO test handout, and so on.

8. Project preview and full portfolio preview

Preview is the checkpoint before anything is published.

  1. Project preview shows only the current project in the exact style it will use inside the website.
  2. Full portfolio preview shows the whole generated website: header, brand, background, personal sections, fun facts, projects, AI area, and contact area.
  3. Save all sections before opening the full preview so every project parser runs first.
  4. Click sections in preview to verify navigation, back arrows, close buttons, collapsible overviews, and file links.
  5. Do not publish until the preview matches what you want recruiters to see.

The preview should match the public website output. If it does not, fix the builder/parser before using Apply to site.

9. Mobile and desktop output

The parser saves responsive layout instructions when projects are saved.

  1. Project windows and section windows should fill the website area on mobile.
  2. Clickable panels wrap to new rows when their names are too long.
  3. Images resize within their section without covering text.
  4. Section titles should be shorter than project titles and should not dominate the page.
  5. Contact should remain the last section, with AI placed above it on mobile when needed.

Before publishing, check at least one narrow/mobile-sized preview and one desktop-sized preview.

10. Publishing target and authentication

Publishing target controls where Apply to site can push. A target is saved only after write access is verified.

  1. Open Publishing target.
  2. Enter the GitHub repository URL or owner/repository target.
  3. Add a custom domain only if that website should use one.
  4. Authenticate with GitHub when prompted.
  5. Use Load from target only after authentication succeeds.

GitHub authorization is cached per repository and branch. Daily reuse avoids repeated prompts, and repeated successful authorization can extend trust for that exact target.

11. Save draft, Apply to site, and offline work

These buttons do different jobs.

  1. Save draft writes the current local builder catalog and assets.
  2. Save all sections runs project and section saves before previewing or publishing.
  3. Apply to site requires a saved draft, copies public files into the portfolio mirror, commits, and pushes.
  4. Builder-only files stay in the builder workspace.
  5. If offline, keep editing and saving drafts. Publish when the network and Git target are available.

If Apply to site says Save draft is required, click Save draft first, then try again.

12. App updates and troubleshooting

Use Check updates to manually look for a newer builder.

  1. When an update is available, click Update.
  2. The app downloads the installer, closes the current builder, updates the existing install, and reopens the app.
  3. Updates are treated as in-place repairs of the detected app.
  4. If an old uninstaller fails, setup stops the builder process and continues updating the same folder.
  5. F11 toggles fullscreen. Press F11 again or press Escape to exit fullscreen.

If the app cannot update, close any old OMB Builder process in Task Manager and reopen the app from the desktop shortcut.

13. Safety rules for public websites

Anything pushed to the portfolio repository can become public.

  1. Do not upload private keys, passwords, private tokens, or private lab data.
  2. Use public GitHub repository links only when the repository is safe for recruiters to inspect.
  3. Remove hidden drafts or old files before publishing if they should not be public.
  4. Keep authentication local. The builder should not commit credentials.
  5. Use the portfolio preview to inspect every public link before publishing.

If another user installs the builder, they should connect their own target unless you explicitly authorize access to your target.

14. Recommended build routine

Use this simple routine when you sit down to add or edit portfolio content.

  1. Open the builder and check the current target.
  2. Edit profile, sections, or projects.
  3. For each project, click Save or Save and close.
  4. Click Save all sections.
  5. Open Portfolio preview and test links, files, mobile layout, and section navigation.
  6. Click Save draft.
  7. Click Apply to site only when the preview is correct.

This keeps the local draft, parsed project previews, full preview, and public website aligned.
