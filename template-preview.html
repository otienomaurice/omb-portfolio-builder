1. First setup

Add the identity and visual pieces first: display name, portfolio label, contact intro, email, phone, public profile links, profile image, brand image, resume, and main background image.

Front-page text controls the hero wording. Fun facts are short personality lines shown near the top of the generated page. Keep them concise so they do not push projects too far down.

Use Save draft after setup. Autosave helps protect work, but Save draft is the deliberate checkpoint before previewing or publishing.

2. Project categories and project creation

All project creation starts from the main Add project control. Choose or create the category first, then select appearance, placement, and title in the create-project window.

Appearance is visual only. It controls background, accent, hover response, click behavior, and panel style. It must not inject predefined text, tools, tests, or fake project content.

Only categories with projects show project rows. Empty categories remain simple headers until a project is added.

3. Project structure

New projects include Overview and Design by default. Additional sections are added manually. Analog projects can also use Simulation as its own major section, separate from Design.

Sections and subsections should contain only real content. Empty sections are not rendered in the final project output. An overview appears automatically when it exists and can be collapsed.

Use Save or Save and close inside the project window to run the parser. Until the project is saved, the latest edits are not inserted into the project preview or full portfolio preview.

4. Rich text, links, formulas, and images

The default typing font is Arial. Highlight exact text before changing font, size, color, bold, italic, or underline. Right-click highlighted text or any rich editor area for common editing commands.

Normal URLs are treated as links. Use Add mathematical formula only when you intentionally want a formula block. Links should open as websites unless they are explicitly attached as downloadable files.

Pasted images land below the cursor line. Added images can be visible on the website or hidden as downloadable files. Visible images must be real image files or direct image URLs.

Drag an inserted image to move the same image. Text around the drop position is split and reflowed so the result behaves like a clean document, not duplicated blocks.

5. Files and evidence

Add files inside the section where they belong: documents, tests, PCB evidence, simulation files, source links, references, or custom subsections.

The builder checks file type. PDFs and documents are downloadable resources, not visible images. Direct web URLs and public share links are saved as links.

Use clear titles and captions. Recruiters should be able to tell what each file proves before opening it.

6. Project preview and full preview

Project preview shows only the selected project, in the exact style it will use inside the full generated page. It is for checking one project without the rest of the portfolio around it.

Full portfolio preview shows the complete generated page: header, brand, background, sections, fun facts, projects, AI area, and contact area. It should match the output that Apply to site publishes.

Save all sections before full preview. That runs all project parsers so the preview uses the latest saved project versions.

7. Mobile and desktop output

The parser prepares responsive layout information when projects are saved. It controls section density, image widths, touch targets, and full-screen section windows for phones and PCs.

Use the preview to check both wide and narrow layouts before publishing. Long titles should be shortened when possible so clickable panels stay readable on mobile.

8. Publishing target and authentication

Publishing target stores the repository URL, branch, optional custom domain, and local credential state for the selected target. The builder keeps the previous target until Git write access is verified.

GitHub authorization is cached per repository and branch. Daily reuse avoids repeated prompts, and repeated successful authorization can extend the trust window for that same target.

If authentication fails, the builder remains local. No live files should be changed until the target is authenticated.

9. Save draft, Apply to site, and local-only work

Save draft writes the current local builder catalog. Apply to site requires a saved draft, copies public files into the publish mirror, commits, and pushes the generated output.

Builder-only files stay in the builder workspace. Public files move into the publish mirror only when the builder prepares the site for publishing.

If the machine is offline, keep editing and saving drafts locally. Publish when the network and Git target are available.

10. App updates and troubleshooting

Use Check updates to manually look for a newer builder. When an update is available, the app downloads the installer, closes the current builder, updates the existing install, and reopens the app.

Updates are treated as in-place repairs of the detected app. If an old uninstaller fails, setup stops the builder process and continues updating the same folder instead of asking you to uninstall manually.

Use File, View, and Help menus for common desktop commands. F11 toggles fullscreen; pressing F11 again or pressing Escape exits fullscreen.
