Profile, Branding, And ContactSet the name, contact links, resume, profile photo, brand icon, and public identity layer.

Start here when the builder is blank, when you are setting up a new machine, or when the public website still has placeholder identity details. This information becomes the stable identity layer of the portfolio: the name recruiters see, the contact links they use, and the visual brand that makes the site recognizable.

Think of this step as setting the portfolio header, business card, and public identity before adding project evidence. Projects can change often, but the profile section should stay clean, accurate, and easy to verify.

  1. Open Profile and contact.
  2. Add display name, portfolio label, contact intro, email, phone, GitHub, LinkedIn, and any personal website links.
  3. Add the profile image, brand or trademark image, resume PDF, and main page background image.
  4. Use a professional profile photo and a brand image that remains clear at small icon sizes.
  5. Check that email opens a compose window and phone uses a call link on supported devices.
  6. Save draft after the profile looks correct, then open Portfolio preview to confirm the public layout.
  • Email should use a real address and should not contain extra spaces.
  • Phone should include country code when the audience may use mobile devices.
  • GitHub, LinkedIn, and website fields should be full links when possible.
  • Resume should be a public-safe PDF, not a private editable document.

Good profile text is short and recruiter-facing. Put personality in Fun facts and personal sections, not in the contact labels. The contact panel should answer one question quickly: who is this person and how do I reach them?

Front Page Text And Fun FactsWrite the first message visitors see and add a short human note near the top.

Front page text controls the first message recruiters see before they click into projects. It should explain the portfolio in plain language: what kind of engineering work is shown, what strengths the site is meant to prove, and why the visitor should keep exploring.

Fun facts sit near the top of the site and make the portfolio feel human without interrupting the engineering content. They should be short, memorable, and safe for a professional audience.

  1. Use the eyebrow as a short label, such as Hardware Engineering Portfolio.
  2. Use the main title for the strongest one-line identity statement.
  3. Use the supporting copy to explain what the portfolio contains.
  4. Keep Fun facts to one to three short lines.
  5. Save draft, then open Portfolio preview to verify the top of the page.
  • The main title should be confident but not exaggerated.
  • The supporting copy should mention the type of work: analog, embedded, digital, PCB, testing, simulations, or firmware.
  • Fun facts should not push the projects down too far on mobile.
  • If a fun fact feels too casual, move it into a personal section instead.

Example Fun facts: I built a builder to build this portfolio. I enjoy a good argument about soccer tactics, culture, and refereeing. The best fun facts make a recruiter remember you while still keeping the portfolio focused on engineering evidence.

Project Categories And CreationCreate categories, add projects, choose appearance, and keep project lists clean.

All project creation starts from the main Add project control. This keeps project creation consistent and prevents hidden category-specific add buttons from creating projects with incomplete metadata.

Categories are the top-level filing system for engineering work. Use them like a recruiter would scan them: Analog and Mixed Signal, Digital, Embedded, PCB, Verification, Power Electronics, or any other category that describes the work clearly.

  1. Click Add project.
  2. Choose the category or create a new category if the project does not fit the existing ones.
  3. Select appearance. Appearance controls visual style only; it must not insert fake project text.
  4. Choose placement: top or bottom of the selected category.
  5. Type a realistic project title, then create the project.
  • Project titles should describe the actual system or circuit, not just the class or assignment name.
  • Appearance templates only control look and interaction behavior. They should not add project claims or sample text.
  • Empty categories stay available inside the builder but are not published in the portfolio preview or website until they contain at least one saved project.
  • Use placement to keep your strongest or most relevant projects near the top.

After creating a project, open it and fill the Overview, then manually add the evidence sections the project actually needs. A project with only a title should not be published because it gives the visitor no evidence to inspect.

Project Structure And Parser RulesUnderstand sections, subsections, overviews, project preview, and what appears on the website.

Projects are built by the parser from the sections you save. The parser reads typed content, rich text, files, images, links, and section order, then builds the project page in the same style used by the public website.

The goal is to avoid hand-editing website code. You organize the project in the builder, save it, preview it, and let the parser produce the clean recruiter-facing version.

  1. New projects include only Overview by default.
  2. Every other section, such as Design, Simulation, Tests, Tools, PCB Builds, Documentation, References, Results, or Compile Code, must be added manually.
  3. Sections and subsections are nested manually. A section can contain subsections, and each subsection can contain its own child sections when the project needs that depth.
  4. Open a subsection to use its action row, or right-click the subsection link, when you need to edit or delete that subsection.
  5. Section chips stay in the top navigation band. Actions for the selected section stay on the far right of that same row, so the content area is reserved for text, files, images, and previews.
  6. An overview appears automatically when it exists; it is not shown as a separate clickable Overview icon.
  7. Project status is metadata. Right-click the project title and open Details to set Draft, In progress, Completed, or Archived, and to choose whether that status appears on the public portfolio.
  8. Empty sections, empty subsections, and empty files do not render on the website.
  • Use Overview for the short explanation of what the section is about.
  • Use subsections when a section needs separate clickable areas such as schematic design, simulation results, PCB layout, test setup, or code links.
  • Keep section order intentional. The website follows the order saved by the builder.
  • Delete empty placeholder sections before publishing so the project does not feel unfinished.

Use Save to parse the project and update its project preview. Use Save and close when you are done editing that project window. If the project preview does not look right, fix it in the builder before using Save draft or Apply to site.

Rich Text EditingFormat selected text, paste cleanly, use keyboard shortcuts, and keep line breaks predictable.

Rich text fields behave like document editors. Click where you want to type, highlight the exact words you want to edit, then apply formatting to that selection. Formatting should not change an entire paragraph unless the whole paragraph is selected.

The builder stores the text as structured content so the portfolio can render it cleanly later. That means the editor may standardize pasted formatting, but it should preserve intentional choices such as bold text, font size, color, links, hard line breaks, images, formulas, and code blocks.

Front-page, profile, contact, fun facts, and project overview fields now use the same selection-level text behavior. Highlight the exact words you want to edit, then right-click the highlighted text to copy, paste, cut, change font, change size, change color, bold, italicize, or underline where that field supports formatting. Identity and contact fields remain text-only, so image, formula, and code block insertion is intentionally disabled there.

  1. Right-click highlighted text in any builder text field to copy, cut, paste, select all, change font, change size, change color, and apply bold, italic, or underline.
  2. The default typing font is Arial.
  3. Static text such as guide text, titles, and labels can also be copied by highlighting it and right-clicking the selection.
  4. Use Enter for a new paragraph. Pasted line breaks remain line breaks.
  5. Typing and wrapping should not create fake new paragraphs; only hard enters and pasted line breaks do.
  6. Use the Code block command or Paste as code for source snippets so they render differently from general text.
  7. Save project and Save all sections both pass front-page, profile, contact, fun facts, and project rich text into the parsed preview catalog before the website is generated.
  • Use normal body-size text for explanations. Reserve larger text for titles and important callouts.
  • Use bold sparingly for important measurements, tool names, or design decisions.
  • Keep color changes readable on the selected project appearance.
  • If a pasted paragraph looks strange, clear the formatting and reapply only the formatting you need.

If the selection menu disappears, click back inside the rich editor and highlight the text again. Before saving, read the section in the project preview to make sure line breaks, lists, and emphasized words still look natural.

Links, Formulas, And CodePaste URLs, equations, source snippets, and runnable code without confusing one type for another.

The builder distinguishes normal links from mathematical formulas because they behave differently on the public website. A link should take the visitor to a webpage or resource. A formula should render as math and stay readable inside the project explanation.

Use this section when pasting content from ChatGPT, documentation, GitHub, LinkedIn, LTspice notes, equations, source code, or lab references. The builder tries to recognize the content, but you should still check the preview before publishing.

  1. URLs such as linkedin.com, github.com, www links, and https links are treated as links.
  2. Use Add mathematical formula only when you intentionally want a formula block.
  3. Use Code block or Paste as code for C, C++, SystemVerilog, LTspice, Java, JavaScript, Python, and HTML snippets.
  4. Choose Basic paste when you want the builder to clean extra trailing whitespace, or Keep source spacing when indentation is important.
  5. Add Code is for presentation snippets in the portfolio. Compile Code is a separate local workspace for buildable source files, nested folders, terminal output, and command execution.
  6. Links in rich text open as websites on the portfolio unless you attach them as downloadable files.
  7. PDFs, spreadsheets, code files, archives, and documents are saved as downloadable resources.
  8. Image files can be visible on the website or hidden as downloadable evidence.
  • Use full links when possible: https://github.com/user/repo is clearer than only github.com.
  • Use link text that explains the destination, such as GitHub repository, LinkedIn profile, test report, or schematic PDF.
  • Use formulas for equations, transfer functions, gain expressions, timing relationships, or component calculations.
  • Use code blocks for short code examples that should be readable on the website; use files or GitHub links for long source trees.
  • If a link is accidentally treated as a formula, remove the formula block and paste it again as normal text or link content.

When pasting from ChatGPT, review formulas and code before saving. If the content is a link, it should render as a clickable link, not as a formula. If the content is source code, it should render inside a dark code block with language coloring instead of becoming ordinary paragraph text.

Compile Code WorkspaceUse the built-in IDE-style workspace for source files, folders, compiler output, terminal commands, and HDL scope viewing.

Compile Code is for local engineering work inside a project. It is separate from Add Code, which is only for polished source snippets that should appear in the public portfolio. Compile Code can stay empty, but when you use it, treat it like a small project workspace.

The left explorer holds folders and source files. Drag the vertical divider beside Solution Explorer when names need more room. Imported files and imported directories appear in the tree without opening editor tabs; click a file in the tree when you want to view or edit it. The editor opens closeable file views, and the bottom output area contains Console, Messages, Terminal, Input, and Scope so build feedback and run settings stay organized instead of mixing everything into one box.

  1. Right-click the workspace tree to add a source file, create a folder, import several files, import a directory, move a file, or delete a selected file or folder. Compiler checks and tool installation live under the Tools menu, not in the right-click tree menu.
  2. Ctrl-click files in the Solution Explorer to select several files, then right-click one of the selected files and choose Delete selected.
  3. Right-click a folder to delete the folder and the files contained inside it after confirmation.
  4. Use File details only when you need to view or edit the absolute path, source language, HDL role, file name, file type, or save status. It opens as a separate moveable, resizable window so it does not steal editor space.
  5. Use Save Source or Ctrl+S to write the active file into the project workspace. Use Save Source As from the File menu when you need an external copy in another location.
  6. Use Beautify to clean the whole active file for C, C++, Verilog, SystemVerilog, LTspice, Java, JavaScript, Python, HTML, or support text files. Syntax coloring updates while typing, after import, and after beautify.
  7. Use Compile to check or build the active source file.
  8. Use Build project when the language needs several files from the project workspace.
  9. Use Run to execute supported languages after a successful build.
  10. Use Input beside the output tabs to open a separate moveable Program input window for stdin before running code.
  11. Use Simulate for Verilog or SystemVerilog when a testbench is present.
  12. Use Append code to project to place the active source into a selected project section or nested subsection as a formatted code block.
  • Right-click a project section or subsection and choose Open in new window when you want to compare up to four parts of the same project without losing your current place.
  • Console shows compiler and runtime output for the active source.
  • Messages shows save, compile, build, append, tool-check, and simulation progress.
  • Terminal runs local PowerShell-style commands from the selected project compile folder. Type beside the prompt, press Enter, read the output directly below the command, then continue from the next prompt.
  • Input opens stdin for the active source in a separate window so the code editor stays focused on source text.
  • Scope appears for Verilog/SystemVerilog and shows VCD waveforms after simulation.
  • Click a scrollable area before using the mouse wheel there. Solution Explorer, file tabs, editor, Console, Messages, Terminal, and Scope each keep their own scroll behavior.
  • The blue status strip stays fixed at the bottom of the output area while the output panel is resized.
  • Icarus Verilog and vvp are the required HDL simulator tools. If Verilator or GTKWave are installed later, the builder detects them and uses Verilator lint as an extra advisory check.

For HDL, add or mark a Testbench file before simulation. The testbench must call $dumpfile and $dumpvars so the scope has signal-over-time data. In the scope, use signal filter, time zoom, per-signal color, and per-signal radix controls to make debugging readable.

Images And File EvidenceAdd images, PDFs, source files, reports, captions, and download-only evidence in the right place.

Add evidence where it belongs so the final portfolio reads like a technical record, not a file dump. Recruiters should be able to click into a project and understand what was designed, what was tested, which tools were used, and what files support the claims.

Images are best for visual evidence: schematics, PCB layouts, test benches, oscilloscope captures, simulation plots, block diagrams, timing diagrams, microscope photos, assembled boards, or setup photos. Files are best for downloadable proof: PDFs, reports, source files, code archives, simulation decks, datasheets, or design notes.

  1. Paste an image into a rich editor to place it below the cursor line.
  2. Use Add image when you need source selection, rename, caption, alignment, crop, or visible/download-only control.
  3. Drag an inserted image to move the same image; the editor should not duplicate it.
  4. Add PDFs under documents, lab reports, design notes, or references.
  5. Add simulation files under Simulation files and screenshots/plots under Results.
  6. Add source repositories as links, not downloadable text files.
  7. All local project evidence file types are accepted, including .sv, .v, .asc, .raw, .zip, PDFs, images, reports, and source files.
  • Use visible images when the visitor should inspect them immediately.
  • Use download-only images or files when they are supporting evidence but would clutter the page.
  • Rename files and images to meaningful names before saving when the default filename is not clear.
  • Add captions that explain what the image proves, not just what the image is.

Project evidence is saved under docs/project-folder/section-or-subsection/filename. For example, a SystemVerilog design file could be saved as docs/laser-link/design/filter.sv. The upload status shows the generated path after the file is saved. Use titles that explain the evidence: Oscilloscope capture, KiCad PCB layout, LTspice transient result, VCO test handout, measured ripple at load, and so on. Before publishing, click every visible image and file link in the preview so broken or private files do not reach the public site.

Project Preview And Portfolio PreviewCheck a single project or the whole public website before saving or publishing.

Preview is the checkpoint before anything is published. It lets you inspect the generated portfolio exactly the way a visitor will experience it, without pushing changes to GitHub Pages yet.

There are two preview levels. Project preview isolates one project and shows how that project will appear inside the final portfolio. Full portfolio preview shows the whole website, including profile, front page text, fun facts, project categories, personal sections, AI area, and contact section.

  1. Project preview shows only the current project in the exact style it will use inside the website.
  2. Full portfolio preview shows the whole generated website: header, brand, background, personal sections, fun facts, projects, AI area, and contact area.
  3. Save all sections before opening the full preview so every project parser runs first.
  4. Click sections in preview to verify navigation, back arrows, close buttons, collapsible overviews, and file links.
  5. Do not publish until the preview matches what you want recruiters to see.
  • Check that the project title is not repeated unnecessarily.
  • Check that overview text appears by default when it exists.
  • Check that empty sections do not appear.
  • Check that website links open in the browser and local files download only when intended.

The preview should match the public website output. If it does not, fix the builder/parser before using Apply to site. Treat preview as the approval screen for your own work.

Mobile And Desktop OutputMake sure the same portfolio works on phones, laptops, and large desktop screens.

The parser saves responsive layout instructions when projects are saved. The same content must work on a laptop screen, a desktop monitor, and a phone. This is especially important because recruiters may open the resume link from a phone before they ever view it on a computer.

Good mobile output is not just smaller desktop output. It should keep titles readable, controls reachable, images contained, and contact information easy to use.

  1. Project windows and section windows should fill the website area on mobile.
  2. Clickable panels wrap to new rows when their names are too long.
  3. Images resize within their section without covering text.
  4. Section titles should be shorter than project titles and should not dominate the page.
  5. Contact should remain the last section, with AI placed above it on mobile when needed.
  • Use shorter section names when possible so buttons do not become awkward on mobile.
  • Keep images high enough quality to inspect but not so large that they make the page lag.
  • Use captions to explain images because phone screens may show less surrounding context.
  • Make sure back, forward, refresh, and close controls stay reachable at the top of windows.

Before publishing, check at least one narrow/mobile-sized preview and one desktop-sized preview. If something only works on desktop, treat it as unfinished because the public site is meant to be viewed from any normal browser.

Publishing Target And AuthenticationConnect the website repository, verify access, and load from a target only after authorization.

Publishing target controls where Apply to site can push. A target is saved only after write access is verified, because the builder should never publish to a website unless the current user has permission to update that repository.

This protects your portfolio from accidental pushes and also lets another user install the builder for their own site without getting access to your website. The target describes where the public portfolio will live; authentication proves the current user can write there.

  1. Open Publishing target.
  2. Enter the GitHub repository URL or owner/repository target.
  3. Click Authenticate target to verify write access for that repository and branch.
  4. Complete GitHub/Git Credential Manager sign-in only when prompted.
  5. Use Load from target only after authentication succeeds and only when you want to import that website repository into this builder.
  • Use the repository that backs the public website, such as a GitHub Pages repository.
  • Use the correct branch. The builder should not guess a branch that does not exist.
  • Authenticate target does not import content; Load from target does not authenticate a new target.
  • If authentication fails, read the message before retrying; it usually explains whether Git, credentials, branch state, or repository access caused the failure.

GitHub authorization is cached per repository and branch. Daily reuse avoids repeated prompts, and repeated successful authorization can extend trust for that exact target. If the target changes, authenticate again because a new target is a different publishing permission boundary.

Save Draft, Apply To Site, And Offline WorkKnow which save button protects local work and which action publishes the website.

These buttons do different jobs, and using them in the right order prevents lost work. Save draft protects the builder state on the local machine. Apply to site publishes the already-saved public version to the selected website target.

The builder can be used offline for most editing work. Publishing is the only part that needs the network and a Git target with write access.

  1. Save draft writes the current local builder catalog and assets.
  2. Save all sections runs project and section saves before previewing or publishing.
  3. Apply to site requires a saved draft, copies public files into the portfolio mirror, commits, and pushes.
  4. Builder-only files stay in the builder workspace.
  5. If offline, keep editing and saving drafts. Publish when the network and Git target are available.
  • Use Save or Save and close inside a project before relying on the project preview.
  • Use Save all sections before opening the full portfolio preview.
  • Use Save draft before Apply to site.
  • Use Apply to site only after the preview looks correct and the target is authenticated.

If Apply to site says Save draft is required, click Save draft first, then try again. This rule exists so the public website is built from a known saved state rather than from half-finished changes still sitting in an open editor window.

App Updates And TroubleshootingUpdate the installed app, avoid duplicate installs, switch viewing themes, and recover from stale windows.

Use Check updates to manually look for a newer builder. Updates come from GitHub Releases after a new installer is built. Use Preferences to select dark builder mode. When dark mode is active, use the Day view control in the main header to return to the normal light editing view. Dark mode only changes how the builder application is viewed; the portfolio is still built from the saved content colors and selected appearances.

An update should behave like a repair of the existing app, not like a second installation. It should close the current builder, update the existing install location, and reopen the app with the new files.

  1. When an update is available, click Update.
  2. The app downloads the installer, closes the current builder, updates the existing install, and reopens the app.
  3. Updates are treated as in-place repairs of the detected app.
  4. If an old uninstaller fails, setup stops the builder process and continues updating the same folder.
  5. F11 toggles fullscreen. Press F11 again or press Escape to exit fullscreen.
  • If the app says it is up to date, no action is needed.
  • If a release is paused, wait for the next version instead of forcing the old installer.
  • If Windows blocks a process from closing, use Task Manager as Administrator to end the stuck installer or old builder process.
  • If the app opens but looks stale, close it fully and reopen it from the desktop shortcut.

If the app cannot update, close any old OMB Builder process in Task Manager and reopen the app from the desktop shortcut. Avoid installing another copy in a different folder; duplicate copies can make Windows shortcuts and update checks point to the wrong version.

Safety Rules For Public WebsitesKeep passwords, private files, private data, and unsafe evidence out of the public portfolio.

Anything pushed to the portfolio repository can become public. Treat the portfolio as a public engineering document. If a file would be unsafe in a public GitHub repository, it should not be uploaded or published through the builder.

Good portfolio evidence is public-safe, clearly labeled, and useful to a visitor. Private credentials, private lab data, personal identity documents, and unpublished confidential work should stay out of the portfolio.

  1. Do not upload private keys, passwords, private tokens, or private lab data.
  2. Use public GitHub repository links only when the repository is safe for recruiters to inspect.
  3. Remove hidden drafts or old files before publishing if they should not be public.
  4. Keep authentication local. The builder should not commit credentials.
  5. Use the portfolio preview to inspect every public link before publishing.
  • Check resumes for address, references, or personal information you do not want public.
  • Check code files for tokens, API keys, Wi-Fi credentials, private comments, or machine-specific paths.
  • Check PDF reports for instructor-only notes, grades, or private collaborator data.
  • Check images for visible serial numbers, passwords, private tabs, or unrelated desktop content.
  • Use Security report to review app download counts, current publishing authorization status, and whether visitor logging requires Cloudflare analytics.

If another user installs the builder, they should connect their own target unless you explicitly authorize access to your target. The builder should help people create portfolios, but it should not give anyone silent access to your website.

Recommended Build RoutineA simple order for editing, saving, previewing, drafting, and publishing safely.

Use this routine when you sit down to add or edit portfolio content. It keeps the builder state, project previews, full portfolio preview, saved draft, and public website aligned.

The routine is intentionally repetitive because publishing is safer when every project has been parsed and every section has been previewed before Git pushes anything.

  1. Open the builder and check the current target.
  2. Edit profile, sections, or projects.
  3. For each project, click Save or Save and close.
  4. Click Save all sections.
  5. Open Portfolio preview and test links, files, mobile layout, and section navigation.
  6. Click Save draft.
  7. Click Apply to site only when the preview is correct.
  • For small text edits, still preview the affected section before publishing.
  • For file or image additions, click the file or image in preview before publishing.
  • For project edits, use project preview first, then full portfolio preview.
  • For layout changes, check both desktop and mobile-sized previews.
  • For publishing target changes, authenticate before loading or applying site data.

This keeps the local draft, parsed project previews, full preview, and public website aligned. A good stopping point is: project saved, all sections saved, full preview checked, draft saved, and website pushed only after the preview looks ready.
