Builder guide

How to build the portfolio

  1. Create projects from Add new project.

    Click Add new project, choose a category, then complete the separate create-project window. The category controls where the project lives; appearance choices are selected from one shared library. Use Add category when you need a project track beyond the default Analog and Mixed Signal, Digital, and Embedded groups.

  2. Select appearance, placement, and title.

    Appearance only controls background, accent color, hover behavior, click response, and preview styling. It does not add text, tools, files, tests, or placeholder project content. Placement controls where the project appears in the project list.

  3. Edit the default project areas.

    Overview and Design are default areas. Analog projects also include Simulation as a separate default area. Overview uses the rich editor for text, images, and formulas. Analog Design includes Design Overview, Documentation, References, Math / Analysis, PCB evidence, and images.

  4. Add files and custom sections.

    Use Add file, Add image, direct web URLs, or supported share links inside the correct subsection. The builder checks file types so visible images remain real image files, PDFs and documents become downloadable files, and normal website links open as web pages. Custom sections can hold extra subsections, files, links, and notes.

  5. Add top-page fun facts.

    Use the Fun facts panel to type up to three short lines. They render near the top of the website, just below the hero, so recruiters see a bit of personality before scrolling into projects.

  6. Build rich summaries deliberately.

    Click Create overview before typing. The default typing font is Arial. Highlight specific text before changing bold, color, font, or size so formatting applies only to that selection. Right-click in the editor to cut, copy, paste, add local images, add direct image URLs, add formulas, adjust font sizing, and set alignment. Pasted images land below the cursor line, direct image URLs must point to an image file, and PDFs are rejected as visible images.

  7. Move images cleanly inside overviews.

    Drag an inserted image to reposition the existing image instead of creating a duplicate. A blue placement marker shows the cursor-like drop location, and text around that point is split and reflowed around the image so the final website reads like one clean document instead of separate chunks.

  8. Save staged editor changes.

    When a small edit form opens, typed content is staged first. It is not committed to the project until the Save button inside that form is clicked. Cancel leaves the previous content unchanged.

  9. Save the project to run the parser.

    Click Save or Save and close in the project window. This invokes the project parser, which reads the typed inputs and files, then builds the recruiter-facing project with Overview shown by default and other sections opening only when they contain real content. The parser also prepares responsive layout settings for section card density, image width, touch targets, and full-screen project windows so the same project works on phones and PCs.

  10. Use Save all sections before previewing the whole site.

    Save all sections runs the parser across all current projects and closes the active project editor. It prepares the projects area for the full portfolio preview.

  11. Inspect Portfolio preview.

    Portfolio preview opens a full website replica with the header, OMB mark, background, resume, process section, contact section, and parsed projects. Search, filters, and navigation links work inside the preview.

  12. Autosave protects builder work.

    The builder autosaves local edits to the draft catalog so closing the builder should not lose work. Save draft is still available when you want to manually write the current local draft. Editing files stay in the builder workspace; public website files are prepared separately when you publish.

  13. Connect a publishing target before going live.

    Use Publishing target to choose the website repository and custom domain. Click Save target and authenticate so GitHub verifies write access before the target is kept. A successful authorization is remembered for the same repository and branch for the day, and frequent successful authorizations can extend that trust window to 30 days. After authentication succeeds, the builder immediately loads the latest compatible website copy from that target into this local builder; if authentication fails, the previous target remains in place and the builder stays local.

  14. Apply to site publishes when ready.

    Apply to site writes the website data, copies only public website files into the portfolio workspace, commits the site changes, and pushes to GitHub automatically without a prompt. After the push finishes, a result window shows whether it succeeded and displays the Git output.

  15. Install builder updates when prompted.

    When a newer released builder is available, the app opens an update window. Choose Update to download the installer, close the builder, install the new release, and reopen the updated app automatically. Choose Remind me later to postpone it, or Skip this version if you do not want that release. Use Check updates in the top bar when you want to check manually.

Publishing setup
  • Checking local publishing tools...

Publishing uses Git for Windows and Git Credential Manager. Enter a repository, then click Save target and authenticate. A browser or GitHub sign-in window opens only when the saved authorization for this repository and branch has expired. The builder keeps the previous target until GitHub confirms write access to the new target, then it loads the latest compatible website copy from that target into this local builder.
